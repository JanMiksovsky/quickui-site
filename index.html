QuickUI (Quick User Interface) is a framework that simplifies the design, development, and maintenance of web-based user interfaces.

  • Create UI in a markup language that’s HTML + extensions for creating your own tags. This markup is easy to write, read, share, and maintain. As with HTML, a single file keeps the related CSS, DOM elements, and JavaScript behavior together.
  • The tags you create, called controls, cleanly define a portion of a page that can be reused. Controls are modular, composable, and subclassable.
  • Write markup in your favorite IDE, then compile it using the QuickUI tools (on Windows, OS/X, or Linux) into highly legible and debuggable JavaScript classes.
  • Define a programmatic interface for your controls that lets them be manipulated without exposing irrelevant implementation details. This avoids fragile code dependencies that impair reusability and maintainability.
  • QuickUI builds upon jQuery, adding a fully object-oriented and extensible control framework. Use all the goodness of object-oriented design to create well-factored UI code.
  • Create top-level pages in QuickUI, or include QuickUI controls in pages built with other tools.
  • Works with Apple Safari, Google Chrome, Internet Explorer (8/9), and Mozilla Firefox.

The page you are reading now (and this entire site) is built using QuickUI. The following brief demos of various QuickUI aspects each show the live output of a QuickUI control running on this page.

HTML macros

Video: Creating an HTML macro

Define your own tag to stand for a block of HTML content. This content will get inserted wherever you use that tag.

<Control name="Greet"> Hello <i>world!</i> </Control>
<Greet/> Yes! <Greet/>
Keep related markup and CSS right next to each other
Old school HTML pages kept the markup and CSS right next to each other, which was great for simple pages but can’t scale up to today’s complex apps. QuickUI restores this simplicity in a scalable way by letting you define a control in a single file that keeps the control’s markup and CSS rules together. No more hunting around for all the styles defined for a particular component!
<Control name="Tag"> <content> &lt;<span id="Tag_content"/>&gt; </content> <style> { display: inline; font-family: monospace; } </style> … </Control>
Quick markup has <Tag>style</Tag> and <Tag>script</Tag> elements, just like HTML.
Define new controls by composing existing ones
QuickUI controls can be used as building blocks for larger and more complex controls. Snap together some markup, CSS, and JavaScript behavior—and now you have a new class that can be used anywhere you want. See the full SearchBox example.
<Control name="SearchBox"> <content> <input id="terms" type="text"/> <OrangeButton>Search</OrangeButton> </content> … </Control>
<SearchBox/>
Refer to DOM elements as class members
The JavaScript generated by the Quick compiler builds your DOM elements at run-time through jQuery calls. As it does so, it saves references to any element with an ID, storing these as members on your control instance. Your JavaScript can refer to these members directly without the need for run-time searches through the DOM to reacquire exactly the element you want. This works even if another element on the page has the same ID!
<Control name="Counter"> <content> You’ve clicked this <span id="count">0</span> times. </content> <script> Counter.prototype.extend({ initialize: function() { // Points to above span. var $count = this.$count(); this.click(function() { $count.text(1 + parseInt($count.text())); }); } }); </script> </Control>
<Counter/> <Counter/> <Counter/>
Easy property definition

Video: Defining control properties in QuickUI

The QuickUI runtime includes a range of helper functions that make it trivial to define properties on a control class and map the property values to (and from) control DOM element content, CSS styles, and more. Properties can be specified in markup as tag attributes or tag content. Compound property syntax lets the markup for control consumers define rich property values. All proprties are directly accessible as JavaScript class members.

<Control name="Recipe"> <content> <div> Name: <span id="Recipe_name"/> <span id="Recipe_rating"/> </div> <div id="Recipe_content"/> </content> <script> Recipe.prototype.extend({ content: Control.chain("$Recipe_content", "content"), name: Control.chain("$Recipe_name", "content"), rating: Control.property.integer( function(rating) { this.$Recipe_rating().text( "*****".substr(0, rating)); }) }) </script> </Control>
<Recipe name="Lasagna" rating="3"> Quick and easy lasagna </Recipe> <Recipe> <name> Pasta <i>con i fagioli</i> </name> <rating>5</rating> <content> A <b>great</b> Tuscan peasant dish </content> </Recipe>
Subclassing
Quick controls can be easily subclassed to create true prototype-based JavaScript subclasses. These inherit DOM elements, CSS rules, and behavior from their parent classes in a well-defined way. One use for subclassing is to create templates for dialogs, page sections, or complete pages (such as the GalleryPage example). Subclasses generally allow for clean factoring of a user interface into easily maintainable and reusable components.
<!-- Sample dialog template --> <Control name="OrangeDialog"> <prototype> <Dialog> <h1 id="OrangeDialog_title"/> <div id="OrangeDialog_content"/> <div id="OrangeDialog_buttons"/> </Dialog> </prototype> <style> { border: 2px solid orange; … } </style> <script> OrangeDialog.prototype.extend({ buttons: Control.element( "OrangeDialog_buttons").content(), … }); </script> </Control>
<!-- Sample use of template --> <Control name="SampleDialog"> <prototype> <OrangeDialog title="What do you think?"> It's easy to use subclassing to create a new dialog -- or an entirely new template. <buttons> <OrangeButton> Huh </OrangeButton> <OrangeButton> Neat! </OrangeButton> </buttons> </OrangeDialog> </prototype> </Control>
Prevent style rule collisions with style scoping

Video: How style scoping works

Controls on a page are as independent of each other as possible, so you can mix controls freely without concern for collisions in the namespaces for DOM element IDs or CSS rule names. For example, the QuickUI compiler ensures that a control’s CSS only applies to instances of that control: each control defines a corresponding CSS class, and the DOM roots of all that control’s instances are stamped with that CSS class. So if two controls define CSS for a class called, say, “message”, the generated CSS will apply the correct rules to each control.

<Control name="Red"> <div class="message">I’m red</div> <style> .message { color: red; } </style> </Control> <Control name="Green"> <div class="message">I’m green</div> <style> .message { color: green; } </style> </Control>
<Red/> <Green/>
Isolate browser inconsistencies
Getting something working across the mainstream browsers often involves variations in the CSS or JavaScript you need. Bundle up that knowledge in a control so you can reapply it when needed. See the complete Gradient source.
<Control name="Gradient"> <script> … if ($.browser.mozilla) { value = "-moz-linear-gradient(" + position + ", " + startColorString + ", " + endColorString + ")"; } this.css(property, value); … </script> </Control>
<Gradient start="#808080" end="#f0f0f0"/>
